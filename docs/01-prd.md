## 1. 개요

### 1.1 배경

사내 약 50명 규모 조직에서 회의실 사용 일정을 관리하기 위한 예약 시스템이 필요하다. 현재 회의실은 1개이나 향후 최대 10개까지 확장될 수 있다. 일정 충돌과 무단 사용을 방지하고, 누가 언제 회의실을 사용하는지 모두가 확인 가능한 단일 진실 공급원(Single Source of Truth)을 마련한다.

### 1.2 목표

- **누구나 쉽게 회의실을 예약**할 수 있는 직관적 UI 제공 (Google Calendar UX 벤치마킹)
- **예약 충돌 0건** — 시스템 레벨에서 시간 겹침 원천 차단
- **반복 예약 지원** — 정기 회의의 매번 등록 부담 제거
- **권한 기반 운영** — 일반 사용자와 관리자의 권한 명확히 분리

### 1.3 범위

**포함**
- 회원가입 및 이메일 인증
- 회의실 예약 CRUD
- 반복 예약 (RRULE 기반)
- 다중 회의실 지원
- 관리자 예외 등록 워크플로우
- 캘린더 형식 UI

**미포함 (향후 검토)**
- 외부 캘린더(Google/Outlook) 동기화
- 모바일 네이티브 앱
- 알림 (이메일/Slack/Push) — Phase 2 이후 검토
- 회의실 이용 통계/리포트
- QR 코드 체크인

## 2. 사용자 정의

### 2.1 사용자 유형

| 유형 | 권한 | 비고 |
|---|---|---|
| **일반 사용자 (USER)** | 본인 예약 CRUD, 전체 예약 조회 | 사번 50명 대부분 |
| **관리자 (ADMIN)** | USER 권한 + 모든 예약 관리, 예외 승인, 회의실 관리, 사용자 관리 | 1~2명 |

### 2.2 페르소나

**페르소나 1: 일반 직원 김OO**
- 매주 정기 1:1 미팅이 있어 반복 예약을 잡고 싶음
- 가끔 휴가/출장으로 특정 일자만 제외하고 싶음
- 모바일에서도 간단히 빈 시간을 확인하고 싶음

**페르소나 2: 관리자 박OO**
- 외부 손님 미팅으로 4시간 초과 예약 요청을 처리해야 함
- 신규 입사자 계정 생성 및 권한 부여
- 회의실이 추가될 때 시스템에 등록

## 3. 기능 요구사항

### 3.1 인증 및 계정

#### 3.1.1 회원가입

| ID | 요구사항 |
|---|---|
| AUTH-001 | 이메일, 비밀번호, 이름은 필수 입력 |
| AUTH-002 | 부서/팀, 사번, 연락처는 선택 입력 (관리자가 후속으로 채울 수도 있음) |
| AUTH-003 | 비밀번호는 최소 8자, 영문+숫자+특수문자 조합 |
| AUTH-004 | 비밀번호는 argon2로 해싱하여 저장 |
| AUTH-005 | 회원가입 시 6자리 숫자 인증 코드를 이메일로 발송 |
| AUTH-006 | 인증 코드 유효시간 10분, 5회 입력 실패 시 코드 무효화 |
| AUTH-007 | 인증 코드 재발송은 60초 쿨다운 적용 |
| AUTH-008 | 인증 완료 전까지는 로그인 불가 |
| AUTH-009 | 인증 미완료 계정은 24시간 후 자동 삭제 |

#### 3.1.2 로그인 / 세션

| ID | 요구사항 |
|---|---|
| AUTH-010 | 이메일 + 비밀번호로 로그인 |
| AUTH-011 | Access Token (15분) + Refresh Token (14일, HttpOnly Cookie) 발급 |
| AUTH-012 | 5회 연속 로그인 실패 시 계정 30분 잠금 |
| AUTH-013 | 로그아웃 시 Refresh Token 무효화 |
| AUTH-014 | 비밀번호 재설정 — 이메일로 재설정 링크 발송 (1시간 유효) |

#### 3.1.3 권한

| ID | 요구사항 |
|---|---|
| AUTH-015 | 사용자 역할은 USER, ADMIN 두 가지 |
| AUTH-016 | 신규 회원가입은 모두 USER로 생성 |
| AUTH-017 | ADMIN은 다른 사용자의 역할을 변경 가능 |
| AUTH-018 | 최소 1명의 ADMIN은 항상 존재 (마지막 ADMIN 강등 불가) |

## 5. 사용자 시나리오 (대표 흐름)

### 5.1 일반 회원가입

1. 사용자가 가입 페이지에서 이메일/비번/이름/(부서) 입력
2. 시스템이 6자리 인증 코드를 이메일로 발송
3. 사용자가 인증 코드 입력 페이지에서 코드 입력
4. 인증 성공 → 가입 완료 → 자동 로그인 후 캘린더 화면으로 이동

### 5.2 일반 예약 생성

1. 사용자가 캘린더에서 빈 시간 슬롯 클릭
2. 회의실, 시작/종료 시간, 제목, 설명 입력 모달 표시
3. "저장" 클릭 시 시스템 검증 (15분 단위, 4시간 이내, 충돌 없음, 미래 시점)
4. 검증 통과 → 예약 생성 → 캘린더에 즉시 반영

### 5.3 반복 예약 생성

1. 사용자가 일반 예약과 동일하게 입력
2. "반복" 옵션 활성화 → 주기/종료 조건 선택 (예: 매주 월요일, 12주)
3. 시스템이 RRULE 생성 후 미래 회차들을 미리 검증
4. 충돌 회차가 있으면 사용자에게 안내 → "충돌 제외하고 등록" 또는 "취소" 선택
5. 등록 완료 → 시리즈로 묶여 캘린더에 표시

### 5.4 관리자 예외 신청 및 승인

1. 사용자가 5시간짜리 예약 시도 → "4시간 초과 — 관리자 승인 필요" 안내
2. 사용자가 사유 입력 후 신청 → PENDING 상태로 저장
3. 관리자 페이지에 신청 목록 표시
4. 관리자가 승인 클릭 → 시스템이 다시 충돌 검증 → 통과 시 예약 생성, APPROVED 상태로 변경
5. 신청자에게 결과 이메일 발송

## 6. 마일스톤 (개발 단계 개요)

상세 일정은 별도 로드맵 문서에서 정의.

| Phase | 목표 | 예상 기간 |
|---|---|---|
| Phase 0 | 프로젝트 스캐폴딩, Docker 환경, CI 파이프라인 | 3일 |
| Phase 1 | 인증 + 회원가입 + 이메일 인증 | 1주 |
| Phase 2 | 단일 회의실, 단일 예약 CRUD, 캘린더 UI | 1.5주 |
| Phase 3 | 다중 회의실, 권한 기반 기능 제어 | 3일 |
| Phase 4 | 반복 예약 (RRULE) + 예외 일자 | 1.5주 |
| Phase 5 | 관리자 예외 승인 워크플로우 | 1주 |
| Phase 6 | 디자인 폴리싱, 모바일 대응, 테마 변수화 | 1주 |
| Phase 7 | 테스트 보강, 문서 정리, 배포 준비 | 1주 |
